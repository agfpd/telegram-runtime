# 05 — Команды

**Русский** · [English](../05-commands.md)

Команды `telegram-runtime <команда>`. Большинство привязок проще ставить через `iapeer connect telegram`, но рантайм даёт и прямые команды.

## Для оператора

| Команда | Что делает |
|---|---|
| `bot add <bot-username> --token <токен>` | Зарегистрировать бота: проверить токен у Telegram (`getMe`), подтвердить его настоящее `@username` (должно совпасть с переданным), сохранить в `bots/<username>/`. |
| `bot list [--json]` | Показать всех ботов: `@username`, настроен ли. |
| `bot remove <bot-username>` | Удалить бота целиком. |
| `interface human --user-id <id>` | Прописать Telegram-аккаунт человека в профиль текущей папки. |
| `interface bot <bot-username> --peer <агент>` | Привязать бота к агенту по его `@username` (прописать привязку в его профиль). |
| `prepare [--user-id <id>]` | Инициализировать профиль пира в текущей папке, опционально сразу задать Telegram-аккаунт. |
| `doctor [--json]` | Проверка цепочки: профиль, аккаунт человека, настройка и привязки ботов, отсутствие ботов без агента. |

## Подтверждения человеком

Когда пир работает в режиме `gated` (`approval-mode` в iapeer), его блокирующие запросы на подтверждение маршрутизируются человеку через брокер подтверждений демона. Мост — это Telegram-лицо брокера: каждый запрос он рисует карточкой с кнопками **Allow / Deny**, тап резолвит запрос. См. [04 — Возможности](04-возможности.md#подтверждения-человеком).

| Команда | Что делает |
|---|---|
| `approvals [--json]` | Чтение очереди ожидающих подтверждений демона со стороны лица (совпадает с `iapeer approvals`). Только чтение. |
| `notices [--json]` | Чтение доски уведомлений демона со стороны лица — пиры, которые сейчас немы из-за ошибки API. Только чтение по контракту: у уведомления нет резолюции. См. [04 — Возможности](04-возможности.md#уведомления-о-немых-пирах). |
| `onboard-approval` | Показать предложение подключить общий approval-бот (единый Telegram-канал для **faceless**-пиров — без своего бота). |
| `onboard-approval --token <token>` | Провижн approval-бота по токену от `@BotFather`: пишет credential с `role=approval`; карточки faceless-пиров идут туда после перезапуска моста. |
| `onboard-approval --decline` / `--decline --yes` | Отказ от approval-бота. Два касания — двойное предупреждение: без него подтверждения faceless-пиров видны только в хостовом баре (`iapeer approvals`) и CLI, но не в Telegram. Faced-пиры не затронуты. |
| `onboard-approval --status` | Состояние approval-бота: провижен, отклонён или не задан. |

## Системные

Эти команды вызывает iapeer и launchd, не оператор напрямую:

| Команда | Что делает |
|---|---|
| `run` | Основной процесс моста: опрашивает ботов, принимает входящие, читает исходящие envelope'ы, маршрутизирует. |
| `self-config` | Per-peer хук настройки при провижне пира (вызывается ядром iapeer с контекстом пира). |
| `self-install` (запуск без аргументов, `npx @agfpd/telegram-runtime`) | Установка пакета: бинарь на место + манифест рантайма. |

## Через iapeer

На практике подключение идёт одной командой ядра, которая сама вызывает `bot add` + `interface bot` + перезапуск моста:

```bash
iapeer connect telegram <агент> --token <токен>
```
